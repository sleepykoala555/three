1. Introduction

In this project, we design an interactive high-fidelity prototype of a telehealth and appointment scheduling mobile app for the start-up named Almond. First, we present some sketches that we created to brainstorm some ideas, then we move on to the low-fidelity prototype. Afterward, we present the final 5-10 screens of high fidelity prototype of the mobile app.


2. Start-up description & context

Start-up: Almond

Description: Almond is a telehealth application for women’s health. Almond makes it easy to get great ObGyn care fast, both at in-person offices and over telehealth.

Context: The problem with traditional ObGyn practices is that physician time is misused, so patients get slow care. Almond is rebuilding back-office tech that saves physicians time and is hiring a wider range of care providers roles, which allows Almond to deliver better outcomes to patients and reduce the amount of time it takes to get their issues resolved.

4. Wireframe from combined designs

What we’re putting in low-fi:

  1. Home page
  2. Video call (patient side) page
  3. Rating your visit page
  4. Staff page
  5. Scheduling/booking appointment pages
  6. Frequently asked questions page (FAQ) / why almond

Notes for Design Decisions:

  1. After looking at all the sets of sketches, we decided to pick the pages that demonstrate the convenience of Almond, which is that users can have all the information in the app, such as looking up different. providers, setting up virtual calls with their doctors and reading FAQs.
  2. For the home page, we decided to include video call icons so that a user may jump into the call right away. This will be the first thing that the user sees when they enter the app.
  3. For the home page, along with the first video call buttons, we have the “Nearest hospitals” section right below the user’s profile. This way, we cover both remote and in-person appointments through the app, which we thought was appropriate for the problem we are trying to solve (allowing users to get faster care).
  4. The video screen looks the way that it does so that the user has the ability to hand it up when they need to and they get to turn off their camera and microphone the way that a typical facetime user interface does.
  5. After hanging up, we thought the user should be directed to rate the quality of the call they were in. We created a standard rating page where the user evaluates the call on several metrics out of five stars.
  6. On the same page, we also included a “Book again” button so that it is easier for the user to schedule another meeting with the doctor right away.
  7. The FAQ page is meant for the user to view the questions that are most commonly asked and attempt to find the questions that they may have. We formatted it like a typical FAQ page where we have the question and the response to the question right below it.
  8. The scheduling page is meant to give the user more information about the doctor that they are scheduling with. It is formatted like a typical information page (about a doctor), where the user receives information about the person that they are trying to book an appointment with.

Summary of feedback and changes made:

  1. Calling page - Change the “calling…” button to “calling” to avoid confusion
  2. Redesign the Home page - including “upcoming appointments” and faq buttons, and take out the nearest hospitals
  3. Make a “meeting details” page -
    1. The meeting page is going to have more details about the doctor before you join the meeting. At the moment we have the upcoming appointment button on the home screen and when you click on it, it takes you to the video call. We should have something that takes us to the “meet your physician” page and then from there, the user can facetime the physician
  4. Make a personal page that includes “past visits” and records
  5. Make a bottom nav bar - make the middle icon a home button, me profile, providers, and FAQ to ensure consistent navigation between the pages

6. User testing Results & analysis


User testing experiment instructions:

Scenario

What you will be looking at today is an interactive prototype, which means not all buttons/functionalities are clickable, but some are. If you click on the black canvas, you’d see the clickable buttons highlighted in blue. Some things may not work like expected. If something does not work, please explain what you would expect to happen and move on. The prototype today is for a mobile app called Almond. It is a telehealth meeting-scheduling and health application for women’s care. Almond makes it easy to get great ObGyn care quickly, both at in-person offices and over telehealth visits.

Task 1

  1. Imagine you are a patient trying to book an appointment using Almond. First, please find a way to see the list of care providers available. Second, please make an appointment with Dr. Justin Lee on January 25th at 10 a.m., and confirm your appointment.

Task metric:

Did you complete the task successfully?Overall, this task was: very difficult ——— very easy?

Verbal response:

What part of the prototype made the task easy or hard? What buttons or design choices did you like or dislike?

Task 2

  1. Imagine you've already booked an appointment with Dr. Justin Lee, and will now attend it. First, please check the list of your upcoming appointments. Second, please hop on a call with Dr. Justin Lee to attend your scheduled virtual appointment. Third, please hang up the call.

Task metric:

Did you complete the task successfully? Overall, this task was: very difficult ——— very easy?

Verbal response:

Did you successfully complete the tasks? What part of the prototype made it easy or hard? What buttons or design choices did you like or dislike?

Task 3

  1. Please spend a couple of minutes playing around and checking out other functions of the app. Note that if you are not sure what’s clickable, you can always click on the black canvas, and you’d see the clickable buttons highlighted in blue.

Verbal response:

What functionalities did you explore? What problem does the app solve or make easier? How valuable is this solution to you?

Verbal response:

What other functionalities do you wish an app like Almond would provide? How might you benefit from using Almond?


Analysis of results

Task completion

Summary of the subtasks

  1. Make an appointment
  2. Attend the virtual appointment
  3. Explore other functionalities in the app

How users did

  1. For the first task, make an appointment, two users thought it was easy, while one user found the interface unintuitive. Out of the two users who completed the task, one user said that it was not clear that when she clicked on the doctor’s name.
  2. For the second task, attend the virtual appointment, all 3 users were able to complete it.

Types of errors users made

  1. When scheduling an appointment, one users made an error of not clicking on the “Learn more” page to navigate to the doctor’s profile page.
  2. User mixed up the care providers page and meeting page, and got confused since the appointment was already there.

Users’ overall thoughts

Things they liked

  1. Having access to the “History” page was useful
  2. Calendar icon for upcoming appointment was intuitive
  3. Liked being able to see medications
  4. Having a rating page
  5. “I like how everything that you needed is on the home page.”
  6. Two users mentioned that they can see the app being really convenient especially for the women who are not really comfortable with their situation.

Things they didn't like

  1. vocabulary: 1 user mentioned that they wished the vocabulary we used was cohesive. We use multiple words to mean the same thing, which are “meeting” and “appointment”. The user wished that we only used one word to describe the same thing, so that it is less confusing on the app to find the calendar and hop in a meeting with the doctor.
  2. Redundant information: with the nav bar that is on the bottom of the screen, some of the information could get redundant since it was being provided on the home page and the navbar.

Thoughts for future improvement

  1. For the care providers page, one user recommended that we included a button which specified that you wanted to book the appointment explicitly along with a button for more information which would make it easier for the user. Another user wanted to see the distance of the providers, and get the option to book a consultation visit (instead of a formal one).
  2. We can change the color scheme of the home page so that it is easier to distinguish the hierarchy of elements and find specific elements on the page. If we change the color to have only two colors, for example, we expect it will be less confusing and overwhelming to the users on the home page.
  3. We should have more background knowledge about medicine and ObGyn:
    1. One user noticed that the medication names are generic and not brand names, which could trigger different allergies.
    2. Two users mentioned that the providers page should include more information. Such as a doctor’s approach to birth control. It’s very important, especially since politics is involved. It would also be good to have the information around abortion, plan b, iud, mamograms. This would make it easier for a woman to navigate the site and it would better cater to their needs. One user also noted that by including more information with procedures that pertain to women, it would better serve our mission of making women’s healthcare more accessible.
